

# ADR-002: Keep indexing assembly inside `Indexers` for now

## Status
Accepted

## Context
The indexing flow evolved from a simple lexical indexer into a richer pipeline that now includes:

- document preprocessing,
- lexical indexing,
- vector indexing,
- pipeline orchestration,
- and stage-specific document resolution.

To keep the client-facing API simple, these responsibilities are currently assembled behind the `Indexers` factory. This gives callers a small and intention-revealing surface such as:

- `Indexers.lexical(...)`
- `Indexers.vector(...)`
- `Indexers.lexicalAndVector(...)`

This approach keeps the public API clean and avoids exposing the internal pipeline mechanics to the client code.

At the same time, we recognize that this design introduces technical debt. The `Indexers` factory currently contains not only factory methods but also several internal pipeline-related implementations. While cohesive for now, this concentration of responsibilities may eventually grow into an oversized assembly file.

## Decision
For the current stage of the project, we will keep the pipeline assembly and its helper types inside `Indexers`.

This is an intentional and pragmatic decision. At this point, preserving a simple public API is more valuable than prematurely extracting every internal component into its own top-level type.

We explicitly accept the technical debt involved in this choice.

## Consequences
### Positive
- Keeps the client-facing indexing API small and easy to understand.
- Hides pipeline complexity from callers.
- Allows the indexing design to continue evolving without destabilizing the public API.
- Supports rapid iteration while the indexing flow is still being refined.

### Negative
- `Indexers` now concentrates multiple responsibilities.
- Internal indexing stages are less visible and less independently testable as standalone top-level types.
- The file may become harder to maintain if more indexing variants or pipeline hooks are introduced.

## Future direction
In the future, we may move toward a more modular design.

If the indexing flow stabilizes and continues to grow, the internal classes currently nested inside `Indexers` may be extracted into dedicated types. Some of those types may become `public` from a Java language perspective in order to improve package organization and collaboration between internal components, while still remaining outside the module's public API through non-exported internal packages.

In other words, the likely future direction is:

- keep `Indexers` as the main public entry point,
- extract pipeline pieces into dedicated classes,
- and place those classes under non-exported internal packages so they remain infrastructure details rather than primary public API.

This future direction intentionally aligns with Jigsaw's module boundaries: the code can evolve toward a more natural package structure without giving up the encapsulation expected from the system.

## Rationale
This decision balances pragmatism and design clarity.

The current nesting model keeps the indexing API elegant while the architecture is still settling. The team understands that this is not necessarily the final form, and the code should be revisited once the pipeline and indexing stages become stable enough to justify extraction.
