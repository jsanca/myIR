

# ADR-003: Stay in-memory for now and defer persistence to a later stage

## Status
Accepted

## Context
The project currently stores its main indexing structures in memory:

- `Corpus`
- `InvertedIndex`
- `DocumentVectorStore`
- `Vocabulary`

This decision has allowed the system to evolve quickly while validating the core architecture around:

- crawling,
- preprocessing,
- lexical indexing,
- vector indexing,
- sparse similarity,
- and search behavior.

Recent experiments with web crawling and vector search confirmed that the current in-memory approach is sufficient for the scale being exercised right now. The system is currently crawling and indexing on the order of hundreds of pages, which is still a reasonable range for an in-memory implementation during this phase of development.

At the same time, the team recognizes that the current architecture stores multiple representations of the same document in memory, including raw content, normalized content, lexical metadata, postings, and sparse vector representations. This means memory growth is real even if it is not yet a practical bottleneck.

## Decision
We will continue using in-memory storage for the current phase of the project.

We are explicitly not introducing disk-backed persistence yet.

Before moving to persistence, the next architectural step in this area should be observability rather than storage optimization. In other words, before redesigning the system around disk-backed structures, we want to better understand how memory usage grows as corpus size increases.

## Consequences
### Positive
- Keeps the architecture simple while core retrieval and indexing behaviors are still evolving.
- Avoids premature persistence complexity.
- Makes experimentation faster during the current stage of the project.
- Keeps debugging and inspection straightforward.

### Negative
- Memory usage will grow as more documents and representations are stored.
- The current implementation does not yet expose enough observability to quantify growth precisely.
- The system may eventually hit practical limits if crawling scope increases significantly.

## Follow-up direction
The next step in this area is to add lightweight observability around in-memory growth.

Examples of useful future metrics include:

- number of documents in the corpus,
- approximate raw content size,
- approximate normalized content size,
- vocabulary size,
- number of postings,
- number of sparse vectors,
- and total non-zero vector dimensions.

These metrics do not need to provide exact heap accounting. Their main purpose is to reveal growth trends and help decide when persistence or more memory-efficient representations become necessary.

## Future direction
If the project grows beyond the practical comfort zone of the current in-memory design, we may later introduce disk-backed or more compact storage strategies.

That future step should happen only after the current indexing and search architecture has stabilized further and after observability gives us a clearer view of where memory pressure is really coming from.

## Rationale
This decision favors simplicity and learning speed over premature optimization.

The current priority is still improving retrieval quality and validating behavior, not optimizing storage too early. The team understands that in-memory storage is a temporary and bounded choice, but it is the right tradeoff for the present stage of the project.
