## ADR: Corpus statistics publication strategy

### Status
Accepted (temporary pragmatic decision)

### Context
`InMemoryCorpus` currently supports direct document mutation through `add(Document)`. The corpus also maintains aggregate counters such as total document length and documents-with-length, and exposes `CorpusStatistics` as an immutable snapshot used by ranking code.

A recent design discussion considered whether writes should remain directly concurrent inside `Corpus`, or whether the system should move toward a more explicit write model such as an `IndexWriter` / `CorpusWriter` with batching, queuing, and explicit publish boundaries.

### Decision
For now, the project will keep the current pragmatic design:

- `InMemoryCorpus.add(Document)` remains the write entry point.
- Aggregate counters are updated incrementally during writes.
- `CorpusStatistics` is exposed as an immutable snapshot.
- Snapshot refresh is coalesced through a `Debouncer`.
- Synchronization remains in place around the compound write/update operation to preserve consistency between the document map and the aggregate counters.

### Rationale
This decision was chosen because it provides a good short-term balance between correctness, simplicity, and development speed.

At the current stage of the project, the main priority is to keep the indexing flow understandable and stable without introducing a larger writer/session abstraction too early.

The current design ensures:

- a simple public API for experimentation
- correct aggregate counters during replacement writes
- stable snapshot-based statistics for ranking
- reduced snapshot rebuild pressure during ingestion bursts

### Consequences
This approach is intentionally pragmatic, but it has known tradeoffs:

- write semantics remain somewhat implicit
- concurrent writes still rely on synchronization for compound consistency
- snapshot publication is eventual within the debounce window
- the model is less explicit than a dedicated writer/commit architecture

These tradeoffs are acceptable for the current phase of the project.

### Future direction
A future iteration may replace direct corpus mutation with a more explicit write model, likely along one of these lines:

- an `IndexWriter` / `CorpusWriter` abstraction
- a queued write pipeline processed sequentially or in batches
- explicit flush/commit boundaries
- snapshot publication at the end of a write session or batch

That future design would make write ordering, publication boundaries, and snapshot visibility more explicit, and would likely reduce the need for ad hoc synchronization inside `Corpus` itself.
